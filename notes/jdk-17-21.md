
# Modern Java Evolution: JDK 17 & 21 Reference Guide

## Language Evolution: Pattern Matching & Type Safety
*The evolution of how we inspect and deconstruct data.*

### The Progression of Pattern Matching
Pattern matching has evolved from a "nice-to-have" syntax in JDK 17 to a fundamental way of writing logic in JDK 21.

| Feature | JDK 17 Status | JDK 21 Status | Purpose |
| :--- | :--- | :--- | :--- |
| **Sealed Classes** | **Finalized** | - | Controls inheritance hierarchies. |
| **Switch Patterns** | **Preview** | **Finalized** | Matches logic against types/values. |
| **Record Patterns** | - | **Finalized** | Deconstructs data objects directly. |

#### A. Sealed Classes (The Foundation)
Sealed classes allow you to define a closed hierarchy. This is the prerequisite for "Exhaustive Switching"—the compiler can now prove you have handled every possible subclass.

```java
// The hierarchy is strictly controlled
public sealed class Shape permits Circle, Rectangle {}

public final class Circle extends Shape {} // Cannot be extended further
public sealed class Rectangle extends Shape permits Square {} // Can be extended
```

#### B. Pattern Matching & Deconstruction (The Implementation)
In JDK 21, we combine Sealed Classes with **Record Patterns** to write code that is both concise and impossible to break via unhandled types.

```java
record Point(int x, int y) {}

static String describe(Object obj) {
    return switch (obj) {
        case Point(int x, int y) -> "Point at: " + x + "," + y; // Deconstruction!
        case String s             -> "String: " + s;
        case null                 -> "It's null";
        default                   -> "Unknown type";
    };
}
```

## 2. Concurrency Evolution: From Heavy to Lightweight
*The shift from OS-managed threads to JVM-managed virtual threads.*

### The Concurrency Mental Model
Traditional Java concurrency was limited by the "Thread-per-Request" model, where every thread was a heavy Operating System (OS) resource.

#### A. Virtual Threads (JDK 21)
Virtual threads are "user-mode" threads. They are extremely lightweight, allowing a single JVM to run millions of threads instead of thousands.

*   **The Use Case:** High-throughput, I/O-bound workloads (e.g., Web Servers, Database calls).
*   **The Rule:** Do not use Virtual Threads for CPU-intensive tasks (like video encoding); use them for tasks that spend time **waiting**.

```java
// The modern way to handle massive concurrency
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    executor.submit(() -> {
        // This task is incredibly cheap to create and schedule
        performIoBoundWork();
    });
}
```

#### B. Scoped Values (The Context Manager)
As we move to millions of threads, `ThreadLocal` becomes a memory bottleneck. **Scoped Values (Preview)** provide a way to share immutable data across virtual threads safely.

```java
var USER_ID = ScopedValue.newInstance();

try (var scope = ScopedValue.where(USER_ID, "admin_123")) {
    // All code inside this block (and child threads) can access USER_ID
    performAction();
} 
// USER_ID is automatically cleared here, preventing memory leaks
```

## 3. Data & Collection Evolution: Predictability & Structure
*Standardizing how we access and manipulate ordered data.*

### Sequenced Collections (JDK 21)
Before JDK 21, getting the "last element" of a set or a list required different, often clunky, logic. **Sequenced Collections** unify this.

**The Benefit:** A single, predictable API for any collection that has a defined order (`List`, `Deque`, `LinkedHashSet`, etc.).

```java
SequencedCollection<String> list = new ArrayList<>(List.of("First", "Middle", "Last"));

list.getFirst(); // "First"
list.getLast();  // "Last"
list.reversed(); // Returns a reversed view of the collection
```

## 4. Platform & Security Evolution: Hardening the JVM
*Moving away from "magic" internals toward explicit, secure boundaries.*

### A. Strong Encapsulation (JDK 17)
JDK 17 ended the era of "hacking" the JDK. Internal APIs (like `sun.*`) are now strictly off-limits.
*   **Impact:** This improves security and ensures that your code won't break when the JDK updates.
*   **The Fix:** If a library requires internal access, you must explicitly grant it via JVM flags: `--add-opens`.

### B. Deprecation of Security Manager (JDK 17)
The legacy Security Manager is being phased out.
*   **Modern Approach:** Security should be handled at the **Environment Level** (Docker containers, Kubernetes policies, or OS-level permissions) rather than inside the Java code itself.
